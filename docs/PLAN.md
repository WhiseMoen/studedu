# План разработки StudEdu

Рабочий бэклог + зафиксированные решения. Обновляется по мере работы.

---

## Зафиксированные решения (и почему)

| Решение | За | Против / риск | Вердикт |
|---|---|---|---|
| Offline-first: Room — источник правды, Supabase — синк | Мгновенный запуск, работает в метро/вузе без сети | Двусторонний синк сложнее, конфликты правок с двух устройств | Принято. Синк по last-write-wins на `updated_at`, пока устройство одно — риска нет |
| Повторения: RRULE + таблица исключений | Нет дублей строк, перенос/отмена одного вхождения не трогает серию | Генерация вхождений на лету — чуть сложнее код ленты | Принято. dmfs/lib-recur берёт на себя парсинг |
| Расписание вуза — read-only кэш через `ScheduleProvider` | Источник сломается — чинится один адаптер; новый вуз = новый адаптер | Данные могут протухнуть между синками | Принято. Синк по кнопке + раз в день, время последнего синка на экране |
| `enrollments` (ученик × предмет): своя ставка и режим оплаты на каждую пару | Разные предметы/цены у одного ученика; у разных учеников разные цены на один предмет | +1 сущность, +1 шаг при заведении ученика | Принято. Без этого модель не отражает реальность |
| Леджер: цена фиксируется в момент начисления | Смена ставки не переписывает историю; скидки/пробные = ручная правка суммы | Начисление — отдельное действие после занятия | Принято. Кнопка «проведено» в одно касание |
| Отмена занятия = всегда бесплатно | Проще UI и модель, соответствует практике | Если политика изменится — добавить необязательное начисление при отмене | Принято (решение пользователя) |
| Режим оплаты (поурочно / пакет / фикс-месяц) — на уровне enrollment | Реальность: разные ученики платят по-разному | Три ветки логики в начислениях | Принято (решение пользователя) |
| Неоморфизм только на 2–3 акцентах, база — пастельный Material 3 | Доступность: контраст WCAG 3:1, читаемость списков | Меньше «вау», чем сплошной soft UI | Принято. Часы, виджет «до события», FAB |
| Монетизация: хуки сейчас, оплата потом | Не пилить биллинг до подтверждения ценности | Хуки — небольшой оверхед | Принято. `EntitlementProvider` + аналитика; RuStore Billing (15%, СБП) — когда решим продавать |
| Тексты, локаль — только RU | Целевая ниша: студент-репетитор РФ | Закрывает зарубежные сторы | Принято на старте, строки уже в ресурсах — i18n дёшев потом |

## Этап 1 — Главный экран (следующий)

**Тема и дизайн-система**
- [x] Пастельная палитра (светлая + тёмная), радиусы 20–24dp, токены в `ui/theme`
- [x] Неоморфный `Modifier` (двойная тень: свет сверху-слева, тень снизу-справа) + эффект «вдавливания» при нажатии (`pressed`, инверсия теней)
- [x] Hero-обложка ленты «Сегодня»: огромные часы (72sp, моноширинные) на всю верхнюю часть экрана (~85% вьюпорта), дата словами, ближайшее событие; лента открывается прокруткой вниз
- [x] Виджет «до ближайшего события: 1 ч 20 мин»
- [x] Неоморфизм на FAB, «Сохранить», «Личный кабинет»/«СДО» (акценты; текстовые кнопки и чипы не тронуты)
- [x] Все иконки — Feather (compose-icons), `material-icons-extended` удалён из зависимостей

**Домен и данные**
- [x] Генератор вхождений: (events + rules + exceptions, диапазон дат) → список конкретных вхождений
- [x] Детектор конфликтов: пересечение интервалов (generic, sweep line), касание границ — не конфликт
- [x] Репозиторий событий поверх Room (Flow на диапазон дат)
- [x] Хук монетизации: интерфейс `EntitlementProvider` (реализация: всё разрешено)
- [ ] Локальная аналитика: таблица `analytics_events` — отложена до Room v3 (чтобы не плодить миграции по одной)

**UI ленты**
- [x] Лента «Сегодня»: хронологический список карточек с временем (упрощение: без получасовой сетки — вернёмся, если списка не хватит)
- [x] Режимы день / неделя / месяц (сетка месяца, тап по дню → «День»); переключатель и стрелки — прилипшая строка (stickyHeader) над лентой
- [x] Создание события: заголовок, комментарий, дата/время, тип, повторение (день / неделя / 2 недели по дням, «N раз» / «до даты»); ряд дней недели на FlowRow (фикс обрезки «вс» на узких экранах)
- [x] Отмена вхождения серии и удаление события/серии из карточки
- [ ] Перенос вхождения из UI (repo-метод готов, экран — после обкатки)
- [ ] Линия «сейчас» в ленте дня

**Проверка этапа:** создать «матан каждые пн/ср 10:00», перенести одно вхождение, убедиться что остальные не сдвинулись; всё работает без сети.

## Этап 2 — Расписание Политеха (момент истины)

- [x] Интерфейс `ScheduleProvider` (`getSchedule`; поиск групп отложен — группа вводится вручную)
- [x] `MospolytechProvider`: JSON rasp.dmami.ru через JsonElement, вся сеть в одном модуле. РИСК: контракт внешний, звонковая сетка захардкожена — проверяется первой реальной синхронизацией
- [x] Группа в настройках (текстовое поле) + кнопка синка + время последнего синка
- [x] Синк в `university_schedule_cache` (неделя назад + 30 дней вперёд, атомарная перезапись группы)
- [x] Пары в ленте: сиреневые карточки, «тип · аудитория · преподаватель», read-only
- [x] Подсветка конфликтов «занятие × пара» — главная фича приложения
- [x] Fallback: ошибка синка показывается в настройках, кэш и приложение живы

**Скрытие пар**
- [x] Тап по паре → «не хожу»: скрыть предмет / скрыть только тип / приглушить
- [x] Приглушённые пары — полупрозрачные с пометкой «не хожу»; скрытые не рисуются
- [x] Скрытые и приглушённые пары НЕ участвуют в конфликтах — слот свободен для учеников
- [x] Список правил в настройках с удалением (возврат пары)

**Проверка этапа:** неделя реального расписания группы в ленте; личное занятие поверх пары — оба горят красным. После этого — честное сравнение с Google Calendar.

## Этап 3 — Ученики и деньги (расширенный)

**Модель (миграция 0003 + Room v3)**
- [x] Таблица `enrollments`: `student_id`, `subject`, `price_per_lesson`, `billing_mode` (per_lesson / package / monthly), `monthly_fee`, `active`
- [x] `events.enrollment_id`, `lesson_records.enrollment_id`, `payments.enrollment_id` + `payments.lesson_record_id`
- [x] Из `students` убраны `subject` и ставки
- [ ] SQL-миграция 0003 написана — применить в Supabase (панель) перед Этапом 5

**Логика**
- [x] «Проведено» из карточки занятия → `lesson_record` + начисление (сумма редактируема: пусто — по ставке, 0 — пробное; фиксируется навсегда)
- [x] Отмена → без начисления
- [ ] Фикс-месяц: автоначисление 1-го числа (WorkManager) — после обкатки ядра
- [x] Баланс = Σ платежей − Σ начислений; «≈ N занятий предоплачено» при положительном балансе

**UI**
- [x] Список учеников: карточки с предметами, ставками и балансом (долг — красным)
- [x] Карточка ученика: сводка месяца (листается назад), занятий по предметам, начислено/получено
- [x] История: занятия (темы, дз) + платежи одной хронологией
- [x] Быстрое добавление платежа и предмета
- [x] Привязка занятия к «ученик × предмет» в форме создания события
- [x] Заметки к занятию: темы, дз — в диалоге «Проведено»
- [x] Общая статистика по всем ученикам: экран `stats` (route в NavHost) из карточки в настройках — занятия/получено/начислено по каждому ученику за этот и прошлый месяц, сводные итоги наверху

**Проверка этапа:** завести ученика с двумя предметами по разным ценам, провести 3 занятия, принять пакетный платёж, поднять ставку — история не изменилась, сводка месяца сходится вручную.

## Этап 5 — Полировка

- [x] Настройки: группа вуза + ручной синк (сделано в Этапе 2); тёмная тема — системная
- [x] Пустые состояния всех экранов с подсказками
- [ ] Переключатель темы (светлая/тёмная/системная)
- [ ] Авторизация Supabase (email magic link) + первый двусторонний синк — после проверки продукта
- [x] Напоминания — реализованы раньше плана, по прямому запросу пользователя (см. ниже)

**Уведомления (v1)**
- [x] Дефолтные интервалы: дедлайны (задачи с `due_date` + события типа DEADLINE) — за месяц/2 недели/неделю/2 дня/в день (09:00 для задач без времени); занятия — за 30 мин и в момент начала; обычные события — за день
- [x] Мьют целиком на конкретное событие/задачу (иконка Bell/BellOff у задачи-дедлайна, пункт «Отключить уведомления» в действиях события). Тонкая настройка «какие именно тиры включить» — отложена: нет экрана редактирования события, вешать не на что
- [x] `AlarmManager.setExactAndAllowWhileIdle` + фолбэк на неточный `set`, если нет права на точные будильники (запрос права — кнопка в «Настройках» на API 31+)
- [x] Планировщик — скользящее окно 35 дней, пересчитывается при любой правке события/задачи + раз в сутки `WorkManager` (окно не протухает, если приложение не открывали) + при перезагрузке (`BootReceiver`, будильники не переживают ребут)
- [x] `request_code` будильника выводится из id события/задачи (+ старта вхождения для серий) и номера тира, не из даты — отмена работает даже если дата уже изменилась
- [x] Room v3 → v4: `reminders_enabled` в `events`/`tasks`, служебная таблица `scheduled_reminders` (учёт для диффа при пересчёте)
- ⚠️ Не проверено на устройстве/эмуляторе в этой сессии (нет доступа к ADB/эмулятору) — прогнать руками: создание/срабатывание напоминания, перезагрузка эмулятора, запрос POST_NOTIFICATIONS на первом запуске

**Дизайн-пакет №3 (фидбэк по неоморфизму/анимациям)**
- [x] `Modifier.neumorphic`: переход pressed-состояния анимированный (было мгновенное), тени рисуются на software-`Bitmap` — `BlurMaskFilter` не блюрит на аппаратном канвасе Compose и давал чёткий («пиксельный») край
- [x] Неоморфизм снят с кнопок «Личный кабинет»/«СДО»/«Сохранить» в настройках и с FAB на «Сегодня»; на «Сегодня» неоморфизм остаётся на часах и добавлен рамкой на сетку месяца
- [x] Часы: цифры с эффектом гравировки (дублируются в цветах теней со сдвигом+блюром) и жирнее (`FontWeight.Black`)
- [x] Переходы между вкладками и в «Статистику» — горизонтальный слайд по порядку вкладок вместо мгновенной смены
- [x] День/Неделя/Месяц — один переключающийся чип вместо трёх (высвобождает место), «Сегодня» больше не переносится на 2 строки (`maxLines = 1`)

## После ядра (по фидбэку от жизни)

- Виджет на домашний экран (Glance): ближайшее событие + конфликты
- Экспорт отчёта по ученику (CSV / картинка для родителей)
- Напоминания об оплате ученикам (шаблон в TG/WhatsApp)
- Статистика дохода за произвольный период, графики
- Второй вуз (новый `ScheduleProvider`) — проверка расширяемости
- Монетизация: RuStore Billing, freemium-границы через `EntitlementProvider`

## Вне скоупа (осознанно)

Канвас заметок, авто-кластеризация, парные занятия (один слот — два ученика), мультивалютность, iOS/KMP, пуш-сервер.

## Тех-долг (осознанный, не блокирует)

- `kotlinOptions` в Gradle помечен deprecated — заменить на `kotlin { compilerOptions { } }` при следующем обновлении Kotlin, не срочно.
- `EventDao.observeEventsAround` выбирает все повторяющиеся события без фильтра по дате окончания серии — при сотнях событий не проблема; если станет узким местом, добавить денормализованное поле `series_end_at`.
- Синк Supabase: Room-поле `group_name` ↔ Postgres-колонка `"group"` — маппинг имён живёт в слое синка (Этап 5).

## Известные риски

1. **Источник расписания Политеха** — не наш, может меняться. Митигация: всё в одном адаптере, кэш переживает поломку источника.
2. **Retention** — конкурент не Zutor, а привычка к Google Calendar. Митигация: месяц личного использования после этапа 2, решение по монетизации только после.
3. **Двусторонний синк** — источник багов. Митигация: пока одно устройство, синк простой (push-only + pull при старте), усложнять только при реальной необходимости.
4. **Неоморфизм и доступность** — только акценты, контраст текста всегда ≥ 4.5:1, интерактив дублируется цветом.
