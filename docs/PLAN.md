# План разработки StudEdu

Рабочий бэклог + зафиксированные решения. Обновляется по мере работы.

---

## Зафиксированные решения (и почему)

| Решение | За | Против / риск | Вердикт |
|---|---|---|---|
| Offline-first: Room — источник правды, Supabase — синк | Мгновенный запуск, работает в метро/вузе без сети | Двусторонний синк сложнее, конфликты правок с двух устройств | Принято. Синк по last-write-wins на `updated_at`, пока устройство одно — риска нет |
| Повторения: RRULE + таблица исключений | Нет дублей строк, перенос/отмена одного вхождения не трогает серию | Генерация вхождений на лету — чуть сложнее код ленты | Принято. dmfs/lib-recur берёт на себя парсинг |
| Расписание вуза — read-only кэш через `ScheduleProvider` | Источник сломается — чинится один адаптер; новый вуз = новый адаптер | Данные могут протухнуть между синками | Принято. Синк по кнопке + раз в день, время последнего синка на экране |
| `enrollments` (ученик × предмет): своя ставка и режим оплаты на каждую пару | Разные предметы/цены у одного ученика; у разных учеников разные цены на один предмет | +1 сущность, +1 шаг при заведении ученика | Принято. Без этого модель не отражает реальность |
| Леджер: цена фиксируется в момент начисления | Смена ставки не переписывает историю; скидки/пробные = ручная правка суммы | Начисление — отдельное действие после занятия | Принято. Кнопка «проведено» в одно касание |
| Отмена занятия = всегда бесплатно | Проще UI и модель, соответствует практике | Если политика изменится — добавить необязательное начисление при отмене | Принято (решение пользователя) |
| Режим оплаты (поурочно / пакет / фикс-месяц) — на уровне enrollment | Реальность: разные ученики платят по-разному | Три ветки логики в начислениях | Принято (решение пользователя) |
| Неоморфизм только на 2–3 акцентах, база — пастельный Material 3 | Доступность: контраст WCAG 3:1, читаемость списков | Меньше «вау», чем сплошной soft UI | Принято. Часы, виджет «до события», FAB |
| Монетизация: хуки сейчас, оплата потом | Не пилить биллинг до подтверждения ценности | Хуки — небольшой оверхед | Принято. `EntitlementProvider` + аналитика; RuStore Billing (15%, СБП) — когда решим продавать |
| Тексты, локаль — только RU | Целевая ниша: студент-репетитор РФ | Закрывает зарубежные сторы | Принято на старте, строки уже в ресурсах — i18n дёшев потом |

## Тех-долг (осознанный, не блокирует)

- `kotlinOptions` в Gradle помечен deprecated — заменить на `kotlin { compilerOptions { } }` при следующем обновлении Kotlin, не срочно.
- `EventDao.observeEventsAround` выбирает все повторяющиеся события без фильтра по дате окончания серии — при сотнях событий не проблема; если станет узким местом, добавить денормализованное поле `series_end_at`.
- Синк Supabase: Room-поле `group_name` ↔ Postgres-колонка `"group"` — маппинг имён живёт в слое синка (Этап 5).
- `ReminderScheduler.refresh()` пересчитывает весь 35-дневный оконный план целиком на каждое мутирующее действие (пометить занятие, тоггл задачи) — для личного объёма данных не проблема, но без дебаунса/инкрементальности; если станет заметно — дебаунсить вызовы вместо пересчёта на каждый чих.
- `StatsViewModel` фиксирует границы «этот/прошлый месяц» один раз при создании ViewModel — теоретически может показать старые границы, если держать экран открытым через полночь на стыке месяца; не пересоздаётся раньше, чем закрыть и открыть «Статистику» заново. Низкий риск, не трогали.

## Ревизия кода 2026-07-15 (полный аудит + правки)

По прямому запросу владельца проведён полный аудит кодовой базы (data/domain/notifications + UI/ViewModel слои) двумя параллельными агентами, затем применены находки:
- **Bitmap в `Modifier.neumorphic` пересоздавался почти на каждом кадре** анимации нажатия (размер битмапа считался от анимированного, а не статичного смещения) — исправлено, кэш теперь стабилен.
- **Гонка при холодном старте уведомлений**: `AppGraph.init()`, `BootReceiver`, суточный `WorkManager` могли одновременно дёрнуть `ReminderScheduler.refresh()` — добавлен `Mutex`, вызовы теперь безопасно сериализуются; сам fire-and-forget вызов из `init()` убран (ненадёжен внутри `BroadcastReceiver` — процесс может умереть раньше корутины), явный вызов перенесён в `MainActivity` через `lifecycleScope`.
- **Дублирование конструирования `PaymentEntity`** в 4 методах `StudentsRepository` — вынесено в приватный `recordPayment(...)`.
- **`AppGraph.eventRepository`/`studentsRepository` были мертвы** — `TodayViewModel`/`StudentsViewModel`/`TasksViewModel` создавали свои копии репозиториев через `AppDatabase.get(context)` вместо переиспользования `AppGraph`. Все три фабрики переведены на `AppGraph` (закрывает старый тех-долг «фабрики перевести на AppGraph»).
- **Ручное суммирование баланса и полная выгрузка истории платежей** в `StudentsViewModel` ради двух чисел (баланс, «месяц оплачен») — заменены на точечные SQL-запросы (`StudentDao.observeBalance`, новый `observeMonthCoveredEnrollmentIds`).
- **Баннер «разреши точные будильники»** в настройках не обновлялся при возврате из системных настроек — добавлен `LifecycleEventObserver` на `ON_RESUME`.
- **Двойной тап на «Сохранить» в диалоге платежа** мог создать два платежа подряд (особенно опасно для «Месяца») — добавлен флаг `submitted`, блокирующий повторный сабмит.
- Тройное дублирование `formatMoney`/`filterMoney`/`formatEditableAmount` (Students/Stats/Settings) и форматирования времени `"%02d:%02d"` (Today/EventEditorSheet) — вынесены в `ui/util/Money.kt` и `RussianDates.time(...)`.
- Удалён мёртвый код: `EventDao.observeEventsForStudent/getRuleById/deleteRule/getExceptionsForEvent/deleteException`, `StudentDao.deleteLessonRecord/deletePayment`, `HiddenLessonDao.deleteById`, `ScheduleCacheDao.lastSyncedAt` (время синка уже трекается через `AppSettings`) — нигде не вызывались.
- SQL двух версий (Flow/suspend) запроса `EventDao` «события вокруг диапазона» вынесен в общую константу `EVENTS_AROUND_SQL`, чтобы не расходились при правках.

- [x] Иконка приложения: векторная монограмма «S» (толстый скруглённый штрих) с эффектом
  гравировки, как у часов на «Сегодня»; фон — `InkPrimary`. Добавлена монохромная версия
  для тематических иконок Android 13+ (`ic_launcher_monochrome`)
- [x] Фикс задержки уведомлений о занятии («в 9:00» приходило в 10–11): тиры занятия
  (за 30 мин / в момент начала) переведены с `setExactAndAllowWhileIdle` на
  `AlarmManager.setAlarmClock` — не подчиняется Doze/App Standby вовсе, в отличие от
  первого (тот на части бакетов/прошивок всё равно откладывается на час-два). Дедлайны/
  личные события не трогали — там точность до дня, задержка на час некритична. Плюс
  баннер в настройках с запросом отключить оптимизацию батареи (вторая частая причина
  на некоторых прошивках)
- [x] `StudentTint` (Room v5→v6, `students.color_tint`, nullable): 6 пастельных свотчей
  (Небо/Роза/Янтарь/Мята/Песок/Сланец), выбираются в диалоге «Изменить ученика»,
  переопределяют обычный `LessonPastel` для карточек занятий этого ученика
- [x] `ScheduleCardView` и новые блоки в таймлайне: вместо `Card` со сплошной заливкой —
  `Surface` в тон фона + `Modifier.neumorphic` (как сетка месяца), цвет типа/оттенок —
  лёгкой подложкой (альфа ~0.3, приглушённее для «не хожу»/прошедших). Объём даёт тень,
  а не контраст цвета — карточки больше не «выпадают» из общего дизайна
- [x] Режимы «Неделя»/«День» переведены с плоского списка карточек на свой таймлайн-грид
  (`TimelineGrid.kt`, без библиотек): часы 0–24 по вертикали, дни неделей — колонками
  сверху, всё в неоморфной рамке как у сетки месяца; внутренний вертикальный скролл
  (авто-подскролл к текущему часу минус 2 при открытии), линия «сейчас», пересекающиеся
  события раскладываются по «дорожкам» (`layoutLanes` — кластеризация по пересечению +
  жадное распределение по первой свободной дорожке, общий подход большинства календарей)
- ⚠️ Не проверено на эмуляторе/устройстве в этой сессии (нет доступа к ADB) — собирается
  и проходит юнит-тесты, но вложенный вертикальный скролл таймлайна внутри внешней ленты
  и раскладка по дорожкам нужно прогнать руками: разное число пересекающихся событий,
  очень короткие (<20 мин) события, недельный грид на узком экране (7 колонок + шкала часов)

## Известные риски

1. **Источник расписания Политеха** — не наш, может меняться. Митигация: всё в одном адаптере, кэш переживает поломку источника.
2. **Retention** — конкурент не Zutor, а привычка к Google Calendar. Митигация: месяц личного использования после этапа 2, решение по монетизации только после.
3. **Двусторонний синк** — источник багов. Митигация: пока одно устройство, синк простой (push-only + pull при старте), усложнять только при реальной необходимости.
4. **Неоморфизм и доступность** — только акценты, контраст текста всегда ≥ 4.5:1, интерактив дублируется цветом.
