# План разработки StudEdu

Рабочий бэклог + зафиксированные решения. Обновляется по мере работы.

---

## Зафиксированные решения (и почему)

| Решение | За | Против / риск | Вердикт |
|---|---|---|---|
| Offline-first: Room — источник правды, Supabase — синк | Мгновенный запуск, работает в метро/вузе без сети | Двусторонний синк сложнее, конфликты правок с двух устройств | Принято. Синк по last-write-wins на `updated_at`, пока устройство одно — риска нет |
| Повторения: RRULE + таблица исключений | Нет дублей строк, перенос/отмена одного вхождения не трогает серию | Генерация вхождений на лету — чуть сложнее код ленты | Принято. dmfs/lib-recur берёт на себя парсинг |
| Расписание вуза — read-only кэш через `ScheduleProvider` | Источник сломается — чинится один адаптер; новый вуз = новый адаптер | Данные могут протухнуть между синками | Принято. Синк по кнопке + раз в день, время последнего синка на экране |
| `enrollments` (ученик × предмет): своя ставка и режим оплаты на каждую пару | Разные предметы/цены у одного ученика; у разных учеников разные цены на один предмет | +1 сущность, +1 шаг при заведении ученика | Принято. Без этого модель не отражает реальность |
| Леджер: цена фиксируется в момент начисления | Смена ставки не переписывает историю; скидки/пробные = ручная правка суммы | Начисление — отдельное действие после занятия | Принято. Кнопка «проведено» в одно касание |
| Отмена занятия = всегда бесплатно | Проще UI и модель, соответствует практике | Если политика изменится — добавить необязательное начисление при отмене | Принято (решение пользователя) |
| Режим оплаты (поурочно / пакет / фикс-месяц) — на уровне enrollment | Реальность: разные ученики платят по-разному | Три ветки логики в начислениях | Принято (решение пользователя) |
| Неоморфизм только на 2–3 акцентах, база — пастельный Material 3 | Доступность: контраст WCAG 3:1, читаемость списков | Меньше «вау», чем сплошной soft UI | Принято. Часы, виджет «до события», FAB |
| Монетизация: хуки сейчас, оплата потом | Не пилить биллинг до подтверждения ценности | Хуки — небольшой оверхед | Принято. `EntitlementProvider` + аналитика; RuStore Billing (15%, СБП) — когда решим продавать |
| Тексты, локаль — только RU | Целевая ниша: студент-репетитор РФ | Закрывает зарубежные сторы | Принято на старте, строки уже в ресурсах — i18n дёшев потом |

## Этап 1 — Главный экран (следующий)

**Тема и дизайн-система**
- [ ] Пастельная палитра (светлая + тёмная), радиусы 20–24dp, токены в `ui/theme`
- [ ] Неоморфный `Modifier` (двойная тень: свет сверху-слева, тень снизу-справа; «вдавливание» при нажатии)
- [ ] Крупные часы на шапке (неоморфизм-акцент), дата: «четверг, 9 июля» — kotlinx-datetime, русская локаль
- [ ] Виджет «до ближайшего события: 1 ч 20 мин»

**Домен и данные**
- [ ] Генератор вхождений: (events + rules + exceptions, диапазон дат) → список конкретных вхождений
- [ ] Детектор конфликтов: пересечение интервалов двух событий → пометка обоих
- [ ] Репозиторий событий поверх Room (Flow на диапазон дат)
- [ ] Хук монетизации: интерфейс `EntitlementProvider` (реализация: всё разрешено)
- [ ] Локальная аналитика: таблица `analytics_events`, помощник `track("event_created")`

**UI ленты**
- [ ] Лента «Сегодня»: карточки по получасовой сетке, текущее время — линия
- [ ] Режимы день / неделя / месяц (переключатель в шапке)
- [ ] Создание события: заголовок, комментарий, время, тип, повторение («каждый день», «каждые N недель по дням», «до даты» / «N раз»)
- [ ] Редактирование вхождения серии: диалог «только это / всю серию» (→ exception / правка правила)
- [ ] Отмена вхождения (свайп или из карточки)

**Проверка этапа:** создать «матан каждые пн/ср 10:00», перенести одно вхождение, убедиться что остальные не сдвинулись; всё работает без сети.

## Этап 2 — Расписание Политеха (момент истины)

- [ ] Интерфейс `ScheduleProvider` (`getGroups`, `getSchedule`)
- [ ] `MospolytechProvider`: разбор JSON-источника (референс — контракты Edugma), вся сеть внутри модуля
- [ ] Выбор группы в настройках (поиск по подстроке)
- [ ] Синк в `university_schedule_cache`: по кнопке + не чаще раза в день; badge времени последнего синка
- [ ] Пары в ленте: визуально отличимы от личных событий (цвет/иконка), read-only
- [ ] Подсветка конфликтов «занятие × пара» — главная фича приложения
- [ ] Fallback: источник недоступен → показываем кэш + предупреждение, приложение не падает

**Проверка этапа:** неделя реального расписания группы в ленте; личное занятие поверх пары — оба горят красным. После этого — честное сравнение с Google Calendar.

## Этап 3 — Ученики и деньги (расширенный)

**Модель (миграция 0002 + Room v2)**
- [ ] Таблица `enrollments`: `student_id`, `subject`, `price_per_lesson`, `billing_mode` (per_lesson / package / monthly), `monthly_fee`, `active`
- [ ] `events.enrollment_id`, `lesson_records.enrollment_id`, `payments.enrollment_id` + `payments.lesson_record_id`
- [ ] Из `students` убрать `subject` и ставки
- [ ] Синк новых таблиц с Supabase

**Логика**
- [ ] «Проведено» одним касанием из ленты → `lesson_record` + начисление по текущей ставке (сумма редактируема: скидка, пробное 0₽)
- [ ] Отмена → без начисления, помечается в истории
- [ ] Фикс-месяц: автоначисление 1-го числа (WorkManager)
- [ ] Баланс = Σ платежей − Σ начислений; для пакетников «≈ N занятий предоплачено»

**UI**
- [ ] Список учеников: карточки с балансом (долг — красным) и ближайшим занятием
- [ ] Карточка ученика: предметы-enrollments, сводка «этот / прошлый месяц» (занятий по предметам, начислено, получено, баланс), листается назад по месяцам
- [ ] История: занятия + платежи одной хронологией; тап по начислению → занятие
- [ ] Быстрое добавление платежа (сумма, дата, комментарий)
- [ ] Общая статистика на вкладке: доход месяц/прошлый, разбивка по ученикам и предметам
- [ ] Заметки к занятию: темы, дз

**Проверка этапа:** завести ученика с двумя предметами по разным ценам, провести 3 занятия, принять пакетный платёж, поднять ставку — история не изменилась, сводка месяца сходится вручную.

## Этап 4 — Заметки + дедлайны (минимум)

- [ ] Список задач: текст, чекбокс, теги, `due_date`
- [ ] Группировка по тегам, фильтр «только дедлайны»
- [ ] Дедлайны с датой ≤ 7 дней подсвечиваются в ленте «Сегодня»

## Этап 5 — Полировка

- [ ] Настройки: вуз/группа, тема (светлая/тёмная/системная), синк вручную
- [ ] Авторизация Supabase (email magic link) + первый двусторонний синк
- [ ] Напоминания WorkManager: за N минут до занятия/пары
- [ ] Пустые состояния и onboarding (первый запуск: «добавь группу и первого ученика»)

## После ядра (по фидбэку от жизни)

- Виджет на домашний экран (Glance): ближайшее событие + конфликты
- Экспорт отчёта по ученику (CSV / картинка для родителей)
- Напоминания об оплате ученикам (шаблон в TG/WhatsApp)
- Статистика дохода за произвольный период, графики
- Второй вуз (новый `ScheduleProvider`) — проверка расширяемости
- Монетизация: RuStore Billing, freemium-границы через `EntitlementProvider`

## Вне скоупа (осознанно)

Канвас заметок, авто-кластеризация, парные занятия (один слот — два ученика), мультивалютность, iOS/KMP, пуш-сервер.

## Известные риски

1. **Источник расписания Политеха** — не наш, может меняться. Митигация: всё в одном адаптере, кэш переживает поломку источника.
2. **Retention** — конкурент не Zutor, а привычка к Google Calendar. Митигация: месяц личного использования после этапа 2, решение по монетизации только после.
3. **Двусторонний синк** — источник багов. Митигация: пока одно устройство, синк простой (push-only + pull при старте), усложнять только при реальной необходимости.
4. **Неоморфизм и доступность** — только акценты, контраст текста всегда ≥ 4.5:1, интерактив дублируется цветом.
